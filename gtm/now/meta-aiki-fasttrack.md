## Problem

Current GTM framing is split across several valid but incomplete ideas:
- **governance** is too defensive and undersells the outcome,
- **multi-agent orchestration** is too abstract and sounds like infrastructure theory,
- **safe coding** is real and valuable, but by itself sounds narrower than the broader transformation teams actually need.

The risk is that we sell either:
1. a compliance/governance package that feels like overhead, or
2. an orchestration/platform story that sounds premature and expensive.

Neither is the best wedge.

The better wedge is a **working transformation package**:
- take a team or bot installation from ad hoc AI coding usage,
- install a concrete operating system for how coding work gets proposed, reviewed, escalated, and accepted,
- and leave behind measurable improvement in speed, clarity, and safety.

---

## Goal

Define **one umbrella offer** that buyers can understand quickly:

> **Aiki FastTrack installs a production-safe AI coding operating system on one repo/workflow in 14 days.**

That umbrella offer should support multiple delivery modes without fragmenting the narrative:
- **human/team install**,
- **clawbot/plugin-led install**,
- and **hybrid escalation** from bot install into broader team rollout.

## Core thesis

### 1) The thing we sell is not governance; it is operating transformation
Buyers do not primarily want “governance.” They want:
- fewer broken AI-generated patches,
- less review thrash,
- clearer agent/human handoffs,
- and more useful throughput from AI coding.

Governance matters, but as an **embedded property** of the system, not the headline.

### 2) The thing we sell is not “orchestration” as a platform abstraction; it is a working coding system
“Multi-agent orchestration” is implementation language, not buyer language.
The buyer-language outcome is:
- one repo,
- one workflow,
- one bounded install,
- one measurable improvement.

The orchestration story should sit underneath the offer, not in front of it.

### 3) “Safe coding” is the shortest path to trust, and Aiki already provides much of it out of the box
A large part of the Felix lesson is that teams do not want generic AI. They want a **safe operating pattern**:
- explicit role boundaries,
- explicit memory/progress surfaces,
- explicit escalation,
- explicit review and approval boundaries.

Aiki already has a strong native advantage here:
- review loop orientation,
- provenance/replay-friendly posture,
- workflow/task structure,
- and safe-coding posture that can be installed without inventing a separate governance product.

That means the commercial opportunity is not merely “teach people safe coding manually.”
It is:
- **package the Aiki operating pattern,**
- install it quickly,
- and in some cases let software/plugin scaffolding do most of the teaching.

### 4) Clawbot installs may be cheaper to buy and cheaper for us to deliver than human training-heavy packages
This is strategically important.

If Aiki’s safe-coding defaults and teaching artifacts can be embedded into:
- a plugin,
- a starter pack,
- or a clawbot-side install flow,

then we may be able to sell a lower-priced FastTrack entry that:
- requires less human time from us,
- is easier for maintainers to try,
- and still creates a clean owner escalation path.

That could become the cheapest, fastest-buying wedge:
- **not because it is lower value,**
- but because the teaching cost is partly automated.

## The umbrella offer

## Aiki FastTrack

**Definition:**
A bounded transformation package that installs a production-safe AI coding operating system for one repo/workflow in a short window, usually 14 days.

**Primary buyer promise:**
- move from ad hoc AI coding usage to a reliable workflow,
- clarify agent/human roles,
- reduce unsafe or noisy patch behavior,
- and prove the change on real work.

**Implementation truth underneath the promise:**
- multi-agent coordination,
- safe coding defaults,
- task/review structure,
- escalation boundaries,
- owner visibility.

### Headline positioning
Use language like:

> Install a working AI coding system on one repo in 14 days.

or:

> Turn ad hoc AI coding into a production-safe workflow with clear roles, review loops, and measurable outcomes.

Avoid leading with:
- governance,
- control plane,
- orchestration platform,
- AI transformation consulting,
- agent society / autonomous org / similar vague language.

## Offer architecture

The key decision is to keep **one umbrella offer** while allowing multiple delivery modes.
That avoids fragmenting the story while still matching buyer maturity.

### Mode A — Team / Human-led FastTrack install
**Best for:** founder-led or small engineering teams already using AI tools but with messy team behavior.

**What is being sold:**
A 14-day install for one repo/workflow with:
- current-state diagnostic,
- role definition,
- review loop setup,
- human approval points,
- pilot runs,
- and a rollout recommendation.

**Where the effort goes:**
- teaching humans the operating model,
- aligning on boundaries,
- setting ownership,
- tuning workflow habits.

**Why it closes:**
- obvious pain,
- concrete transformation,
- near-term team productivity gain.

**Why it is relatively expensive:**
- meaningful human teaching/change-management load.

### Mode B — Clawbot / plugin-led FastTrack install
**Best for:** existing clawbot/bot installations or maintainers who already have an agent in the workflow.

**What is being sold:**
A lower-friction install that teaches the operating model mostly through:
- plugin scaffolding,
- install manifests,
- stepwise safe-coding defaults,
- owner packet generation,
- and maintainer-facing quickstart/checklists.

**Where the effort goes:**
- packaging Aiki’s safe-coding and workflow model into installable artifacts,
- detecting risk/chaos signals,
- generating improvement recommendations,
- and producing a clear owner escalation path.

**Why this matters:**
This mode may be **cheaper to buy** than human-teaching-heavy FastTrack because:
- less meeting time,
- less live training,
- less organizational alignment cost,
- and more productized delivery.

**Why margins may be better:**
Because a plugin or standardized install flow can do much of the teaching and setup repeatedly.

### Mode C — Hybrid escalation path
**Best for:** accounts that enter through the clawbot/plugin path and then expand into team workflow transformation.

**Pattern:**
1. install clawbot-safe-coding starter,
2. generate owner packet + initial evidence,
3. close a larger 14-day team install,
4. expand into retainer or broader rollout.

This may become the default funnel:
- cheaper entry,
- clearer evidence,
- easier owner buy-in,
- larger downstream project.

## Why this is different from a pure governance package

A pure governance package sounds like:
- rules,
- policy,
- approvals,
- risk reduction.

That is important, but not enough.

FastTrack should instead feel like:
- **you get a working system,**
- with governance built in,
- not a governance binder sitting beside the system.

Governance should be framed as a byproduct of a better operating model.

---

## Why this is different from a generic consulting package

Generic consulting says:
- assess,
- advise,
- recommend.

FastTrack should say:
- install,
- run,
- tune,
- prove.

The buyer should expect actual operating artifacts, such as:
- role map,
- workflow boundaries,
- owner packet,
- review loop config,
- sample runs,
- measurable before/after metrics,
- follow-on rollout recommendation.

## Economic thesis

## Primary revenue logic
The fastest route to near-term revenue is still a bounded transformation offer.
A single well-scoped FastTrack engagement can plausibly land in the low-to-mid four figures or higher, and one or two closes can matter quickly.

## Secondary revenue logic
The plugin-led/clawbot-led mode creates a lower-friction wedge that may:
- convert faster,
- support lighter-weight initial price points,
- and produce follow-on larger installs.

## Important pricing implication
Do **not** think of the clawbot/plugin path as “cheap because it is less valuable.”
Think of it as:
- lower delivery cost,
- lower adoption friction,
- lower need for synchronous teaching,
- therefore potentially lower price **and** strong margin.

The right price strategy is probably:
- team install = premium transformation offer,
- clawbot/plugin install = lower-friction entry offer,
- hybrid expansion = upsell into team install / retainer.

## What the clawbot/plugin-led offer actually needs to do

This is the most important product-design implication from this plan.

If we want a lower-cost FastTrack entry, the plugin/skill/install path must do real teaching, not just setup.

### It should teach:
- what Aiki is controlling,
- what safe coding means in practice,
- what can happen automatically vs what requires review,
- what role the maintainer plays,
- what role the owner plays,
- and what evidence will be generated.

### It should produce:
- current-state workflow/risk summary,
- recommended safe-coding mode,
- role/boundary suggestion,
- sample owner packet,
- installation checklist,
- quickstart for the maintainer,
- and a measurable “before” snapshot.

### It should escalate:
- from read-only observation,
- to draft recommendations,
- to bounded execution,
- to owner-approved expansion.

That is directly aligned with the Felix trust-ladder insight and with Aiki’s native posture.

## Safe-coding-first framing

The strongest strategic insight here is that **safe coding may be the cheapest believable wedge** because it is:
- understandable,
- urgent,
- easy to demonstrate,
- and already a native strength for Aiki.

The transformation offer should therefore be explained as:

> Start with safe coding. End with a working multi-agent coding system.

That lets us tell the truth in sequence:
1. safe coding is the first trust layer,
2. review loop and role clarity come next,
3. broader orchestration becomes credible after that.

This is materially better than trying to sell “multi-agent orchestration” cold.

## Target customer logic

### Best initial customers
- founder-led engineering teams,
- small AI-native product teams,
- devtools companies already experimenting with agents,
- clawbot maintainers/operators,
- service firms running repeated AI coding workflows.

### Best initial signs of pain
- repeated patch/review churn,
- unclear approval boundaries,
- humans redoing AI work constantly,
- multiple agents/tools but no operating model,
- fear of unsafe changes despite clear appetite for AI coding.

### Weak early customers
- buyers who only want compliance paperwork,
- teams not yet using AI coding at all,
- very large enterprise committees before proof exists,
- teams seeking broad platform procurement before a narrow workflow win.

## Proof model

FastTrack must be sold with artifacts, not adjectives.

### Every pilot / install should try to show:
- current workflow shape,
- current failure/rework signal,
- the safe-coding boundaries introduced,
- one or two real runs through the new workflow,
- and a short owner-readable summary of what changed.

### Useful metrics to standardize later
- human intervention count,
- patch acceptance rate,
- review turnaround time,
- repeated regression / rollback indicators,
- number of unsafe actions prevented or escalated,
- operator confidence / clarity improvement.

Not all accounts will have all metrics, but the pattern must remain measurable.

## Delivery design principles

### 1) One repo, one workflow, one bounded outcome
The offer should stay narrow enough to finish and demonstrate.

### 2) Productize the teaching wherever possible
This is especially important for the clawbot/plugin path.
Any repeated explanation should become:
- a checklist,
- a quickstart,
- a config/template,
- or a plugin-generated teaching artifact.

### 3) Keep the owner in the loop for commitment-level actions
The maintainer can install and observe.
The owner signs off on expanded autonomy, commercial scope, or org-level rollout.

### 4) Sell the install, not the infinite journey
The initial offer must feel finite and shippable.
Broader transformation can come after proof.

## Immediate implications

### Strategic choice
The offer family should stay centered on **Aiki FastTrack** as the main name.
That name can hold both:
- premium team installs,
- and lower-friction clawbot/plugin installs.

### Commercial choice
We should explicitly test whether:
- the lower-cost clawbot/plugin-led install closes faster,
- and whether it reliably expands into larger installs or retainers.

### Product choice
We should treat “teaching through the plugin/install flow” as a first-class product surface, not an afterthought.
If this works, it is a real wedge:
- safer by default,
- cheaper to deliver,
- easier to adopt,
- and highly legible to maintainers.

---

## Risks / caveats

### 1) Over-abstracting the story
If we lead with “multi-agent orchestration,” the offer will sound more advanced than it feels useful.

### 2) Under-selling the transformation
If we lead only with “safe coding,” we may miss the broader system change buyers are actually paying for.

### 3) Building too much product too early
The clawbot/plugin teaching path should be productized enough to reduce delivery cost, but not so overbuilt that we delay revenue.

### 4) Scope drift in the premium install
The team install can turn into broad organizational consulting unless the repo/workflow boundary is explicit.

### 5) Confusing free/installable artifacts with the paid offer
The installable starter should create trust and evidence, not give away the entire transformation service.

---

## Recommended stance

For now, the correct stance is:

- **Sell Aiki FastTrack as a transformation install.**
- **Lead with safe coding and working outcomes, not governance or orchestration language.**
- **Design the clawbot/plugin path as a lower-cost, lower-friction entry because Aiki already does much of the safe-coding work natively.**
- **Use that cheaper entry to create evidence, owner packets, and expansion into broader team installs.**

That is a tighter and more believable wedge than either:
- a pure governance package, or
- a cold “multi-agent orchestration” pitch.
