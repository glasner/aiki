# Tu Presence Strategy (Long-Term)

**Owner:** Tu/Jordan  
**Scope:** Tu’s online presence and professional signal, with Aiki as the operating context.

## Purpose
Build a durable, trusted **Tu public persona** in AI execution spaces that is distinct from “Aiki as a whole.”

The goal is to make Tu identifiable as:
- a reliable builder/operator who ships with AI systems,
- a partner who is blunt and evidence-based,
- and a person who consistently reduces risk for teams using AI coding automation.

This is not brand fiction. It is a distribution and reputation strategy for long-term founder-led GTM.

## Core thesis
- **Bots, maintainers, and owners engage when value is operational, measurable, and installable.**
- **Tu’s position is: reliable operator > hype > theoretical advisor.**
- Reputation compounds only if repeated outputs are useful and auditable.

## Positioning pillars
1. **Identity consistency for Tu:**
   - direct, short, evidence-first communication,
   - blunt on risk and tradeoffs,
   - avoids fluff and vague claims.
2. **Artifact-first proof:**
   - each claim has an artifact (runbook, report, policy baseline, sample metrics),
   - if we can’t show results, we don’t claim they’re there.
3. **Memory and iteration discipline:**
   - recurring review loops,
   - explicit guardrails,
   - explicit upgrade path for how Tu operates, not just what Tu sells.
4. **Founder-led bottoms-up motion:**
   - maintainers/bots can trigger initial visibility,
   - owner-level decision still required for paid commitments.

## Presence and channel strategy
- **Primary inbound:** bot-runtime/developer ecosystems where automation tooling is already being consumed.
- **Secondary inbound:** technical notes/postmortems, short operational writeups, and proof snippets.
- **Owner bridge:** every bot-triggered lead must have a short, owner-readable packet with problem → signal → measurable outcome hypothesis.

## Clawbot community outreach motion (HashiCorp-style, bot-first)
Think of this as HashiCorp-style developer motion, except the customer is often the bot maintainer and then the owner.

### 1) Registry-first growth (the equivalent of modules/providers)
- Publish a small set of official installable artifacts:
  - `Aiki FastTrack Human Track`
  - `Aiki FastTrack Clawbot Track`
  - `Aiki Risk Audit`
- Host them in the same ecosystems clawmakers already search:
  - Claw Mart,
  - GitHub docs/README snippets,
  - internal/third-party marketplace templates,
  - and bot runtime registries.
- Each artifact includes versioned install instructions and a standard output contract.

### 2) Evidence-first content loop
- Publish **1 concrete artifact post/week** from actual usage:
  - “before/after risk score,”
  - “failed patch loop reduced from X→Y,”
  - “median sign-off time before/after.”
- Every post must include:
  - a command example,
  - expected outputs,
  - and an owner decision question.

### 3) Maintainer-triggered TOFU (direct-to-bot)
- First touch is always maintainer/bot-facing:
  1. Bot detects risk signal,
  2. auto-generates an owner packet,
  3. posts a low-friction invite to install Aiki FastTrack for pilot.
- This mirrors HashiCorp’s pattern where adoption begins with practical tool usefulness, not pitchy sales copy.

### 4) Owner conversion motion
- Owner packet format (short and repeatable):
  - Problem observed,
  - Current cost/risk,
  - Expected 14-day outcome,
  - Scope + timeline,
  - Pricing band + retainer path.
- Sales message comes **after** maintainer-authored evidence, not before.

### 5) Community growth loop
- Maintain at least 3-5 external templates by the 2nd month.
- Add one maintainer integration each week.
- Track conversion rate from maintainer packet → owner approval and publish results in the journal.

## Lessons imported from Felix's "Hire an AI" playbook

### 1) Position as **operating hire**, not a product
Treat Aiki FastTrack as a working operating pattern:
- persistent memory (`MEMORY.md` + daily logs),
- explicit role boundaries (`SOUL.md` / `IDENTITY.md`),
- and measurable control loops (risk score, interventions, owner packet).

### 2) Trust ladder (governed autonomy)
- **Read-only first:** bot can observe and report.
- **Draft + approval:** bot drafts, human/owner approves before external action.
- **Bounded execute:** only pre-approved, reversible actions.
- **Full autonomy (rare):** only for low-risk repetitive tasks.

This is a differentiator in clawbot motion: every packet and output should map to a rung.

### 3) Security default for operations
- `Email` and any non-verified command channel is treated as **untrusted input**.
- No autonomous action on external instructions.
- Every owner packet contains: trigger signal → risk profile → proposed intervention.

### 4) Repeatable operator artifacts
Keep these as part of core distribution, not optional docs:
- Identity pack (`SOUL.md` / `IDENTITY.md` style profile)
- Evidence pack (`risk-score.schema.json`, sample packet)
- Escalation pack (escalation triggers + fallback behavior)
- Postmortem/post-close summary snippets for proof
