aiki status — v1

Consolidated design: Option A sidebar with plan lifecycle, collapsible Other Tasks, and filtering. Realistic 5–8 word task names with truncation. Lane visualization with five states: complete, running, ready, failed, stuck.
The sidebar

The sidebar is the ops/now/ directory. Plans are markdown files listed individually. Below plans, a single ▸ Other Tasks (3) entry represents all tasks not linked to any plan. It's collapsed by default — just a count. Select it to filter the main pane, or expand it to see task names inline.

Other Tasks don't have the build → review → fix pipeline. They're arbitrary subtask trees with dependencies. When drilled into, they show their DAG directly (no phase labels). Some may have review/fix steps visible if those were triggered, but the structure is a free-form dependency graph, not the three-phase pipeline that plan-linked tasks follow.

DAG nodes render as dots, not names. Real task names are 5–8 words (e.g. "Verify Stripe webhook signature handling") — far too long for inline display between connector chars. Nodes show as ●/◉/○ with short 4-char task IDs in dim. The active/focused node gets a truncated name inline (first 2–3 words + … if needed). Completed nodes show as unlabeled dots. The log area shows IDs too, with names only on the currently active line.

Task name truncation. Names are 5–8 words (30–45 chars). The main pane is ~55 chars wide. In headers, names get their own line — agent + time drop to line 2. In row view, the TASK column is ~22 chars — everything truncates. In the sidebar inline, only ~11 chars fit. Truncation uses … (single-char ellipsis, U+2026).

Plan lifecycle states: ○ draft, ● ready, ◉ editing, ▸ building, ✓ done.

Expanding Other Tasks inline (optional)

Pressing [→] or [l] on the Other Tasks entry expands it to show task names inline in the sidebar. The ▾ chevron indicates expanded state. Press [←] or [h] to collapse. This is a preview — the main pane doesn't filter until you press [Enter].

Truncation in the sidebar. The sidebar has ~11 chars for task names. With 5–8 word names, only the first 1–2 words are visible: "Fix login r…", "Bump all o…", "Refactor e…". This is deliberately minimal — just enough to jog memory about what's there. The main pane shows full names when you select the group.

Row view (Tier 2) — truncation is the norm

At 4+ tasks, auto-switches to row view. The TASK column is ~22 chars wide — every 5–8 word name truncates with …. This is by design: the row view is for scanning status at a glance, not reading full names. Press [Enter] on any row to see the full name in the detail view.

Key difference in row view: Plan tasks show three phase bars (BUILD, REVIEW, FIX). Other Tasks show a single merged progress bar spanning all three columns, since they don't follow the pipeline. The SUBTASKS label replaces phase headers for these rows.

Lane view — parallel build execution

The build phase for a plan task, organized by lanes. Each lane is an independent execution stream with its own agent. Lanes are derived from the subtask DAG at query time — needs-context chains form sessions, depends-on edges define ordering, and fan-out points split into parallel lanes.

Layout. One row per lane. Lane ID (dim 4-char prefix of head task) on the left. DAG dots for the lane's tasks in the middle. Lane status + agent + time on the right. Active lanes show the active node's truncated name. Blocked lanes show ← nmps,kyxr to indicate fan-in dependencies.

Lane xtut (explore → plan) is complete — it was a single session of two needs-context-linked tasks. Lanes nmps and kyxr are running in parallel on different agents. Lane zxrk is blocked, waiting for both implementation lanes to finish (fan-in).

The header shows 4 lanes count and lists the agents driving the task. The log area shows lane-level events: completions, parallel starts.

v1 design decisions

Task names are 5–8 words (30–45 chars)
Real task names like "Migrate payment processing to Stripe" or "Fix login redirect after session timeout" are much longer than placeholder names. The design accounts for this with a consistent truncation strategy at every tier:
- Detail header: Full name on its own line, agent + time on a second line. Names up to ~50 chars display in full; longer names truncate with …
- Row view TASK column: ~22 chars with … truncation. Enough for first 3–4 words. [Enter] to see full name in detail view
- Sidebar inline: ~11 chars with …. Just enough to jog memory ("Fix login r…", "Refactor e…"). The main pane shows full names
- DAG active node: First 2–3 words + … after the dot symbol. Only the focused node gets a label; all others are unlabeled dots
- Log entries: Short IDs by default; active entry shows truncated name

Two-line task cards
With longer names, the previous single-line layout (name + agent + time on one line) gets too cramped. The v1 design uses two-line cards in list views: name on line 1, agent + phase bars on line 2. This gives names full width (~50 chars) and keeps status information scannable. The row view (Tier 2) still uses single-line rows but with aggressive truncation.

Agent abbreviations
To save horizontal space alongside long task names: cc = claude-code, cur = cursor, dev = devin. Full names only appear in drill-down detail views where there's room.

Other Tasks as a single collapsible entry
▸ Other Tasks (3) is one line in the sidebar regardless of how many exist. This keeps the sidebar proportional to plans, not tasks. Three interactions:
- Enter — filter the main pane to Other Tasks (the "list" lives in the main pane)
- → / l — expand inline to preview task names in the sidebar
- ← / h — collapse back to single line

Plan tasks vs Other Tasks — rendering difference
Plan tasks follow the build → review → fix pipeline. Their detail views show three phase sections, and their row views show three phase bars.
Other Tasks are free-form subtask trees with dependencies. They don't have pipeline phases. Their detail views show a raw subtask DAG (no phase labels). Their row views show a single merged progress bar with a completed/total fraction.

DAG nodes are dots, not names
With 5–8 word subtask names ("Verify Stripe webhook signature handling", "Update checkout integration tests for new flow"), inline display is impossible. So:
- All nodes render as state dots: ● done, ◉ active, ○ pending
- Active/focused nodes show a truncated name (first 2–3 words + …) after the dot + short ID
- Completed/pending nodes show only a 4-char task ID in dim
- Log entries show IDs, with name only on the current active line
This keeps the DAG compact and readable at any tree width.

Sidebar as scope selector
Every sidebar item (plan or Other Tasks) is a filter. Selecting one narrows the main pane. The bold/white text on the selected entry plus the "showing X" header make the active filter obvious. [a] shows all, [Esc] deselects. Plans and Other Tasks are symmetrical — same interaction model.

Plan lifecycle in sidebar
Five states: ○ draft, ◉ editing, ● ready, ▸ building, ✓ done. The [b] build action only appears on ready plans. Drafts get [e] edit + [r] mark ready.

Lanes — parallel execution streams
Lanes are derived from the subtask DAG at query time (not stored). Each lane is a sequence of sessions that can run independently. The lane view (scenes 12–15) is the primary detail view for building plans:
- One row per lane — lane ID (dim 4-char prefix of head task) on left, DAG dots in middle, status + agent + time on right
- Active lanes show the active node's truncated name. Completed lanes show all-green dots. Blocked lanes show ← deps and blocked in dim
- Fan-in is shown by the dependency annotation on blocked lanes. When all predecessors complete (Closed(Done) only), the lane unblocks and starts
- Log area shows lane-level events: completions, unblocking, parallel starts
- Header shows lane count and the agents driving the task. When any lane has failed, a red 1 failed count appears in the header
- [l] toggles between lane view and raw DAG — the raw DAG (scene 1) shows the same data without lane organization. Lane view is default for building plans
- Row view [2‖] — the existing parallel notation in row view (scenes 4, 5, 8) represents lane parallelism. [2‖] means "2 lanes active." This was already in the design; lanes formalize what it means

Lane states — five visual states
Lanes are always in exactly one of five states, derived from task statuses at query time:
- Complete ✓ green — all tasks in the lane are Closed(Done). Green lane ID, green dots, green check. Dependent lanes can now unblock
- Running ▸ yellow — at least one task is InProgress. Default lane ID, yellow active dot, agent + time on right
- Ready ● cyan — lane has work, all predecessors complete, next session unblocked by is_blocked(). Cyan lane ID, cyan first dot, ● ready label
- Failed ✗ red — any task is Stopped or Closed(WontDo). Red lane ID, red X on the failed node, ✗ failed label. Dependent lanes will never unblock until intervention
- Stuck stuck red — a special case of blocked: the lane is waiting on a predecessor that failed, not just one that's incomplete. Red stuck label, dim dots, dependency annotation with the failed dep highlighted in red

Stuck vs blocked. Blocked lanes are waiting on work in progress — they'll unblock naturally. Stuck lanes are waiting on a failed dependency — they need intervention (retry, rework, or close as won't-do). The TUI distinguishes these visually: blocked in dim gray vs stuck in red. The header shows a red N failed count when any lane has failed. The key binding bar offers [r] retry lane when a failed lane is focused.

Readiness checks all blocking link types. Lane structure is derived from needs-context + depends-on only. But lane readiness delegates to TaskGraph::is_blocked(), which checks all blocking types (blocked-by, validates, remediates, follows-up). A task blocked by e.g. validates correctly prevents its lane from being ready, even though validates doesn't affect lane structure.

Sessions within lanes
A lane may contain multiple sessions (e.g., needs-context chain → standalone task → another chain). Session boundaries are implicit in the lane view — they're just successive groups of dots. If session visualization becomes important, a ╎ separator between session groups is a natural extension. Deferred for now — single-session lanes are the common case and multi-session lanes still render clearly as dot sequences.
