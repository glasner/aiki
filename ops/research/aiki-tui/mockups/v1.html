aiki status — v1

Consolidated design: Option A sidebar with plan lifecycle, collapsible Other Tasks, and filtering. Plan tasks show build/review/fix pipeline; Other Tasks show free-form subtask dependency trees.
The sidebar

The sidebar is the ops/now/ directory. Plans are markdown files listed individually. Below plans, a single ▸ Other Tasks (3) entry represents all tasks not linked to any plan. It's collapsed by default — just a count. Select it to filter the main pane, or expand it to see task names inline.

Other Tasks don't have the build → review → fix pipeline. They're arbitrary subtask trees with dependencies. When drilled into, they show their DAG directly (no phase labels). Some may have review/fix steps visible if those were triggered, but the structure is a free-form dependency graph, not the three-phase pipeline that plan-linked tasks follow.

DAG nodes render as dots, not names. Subtask names are too long for inline display. Nodes show as ●/◉/○ with short 4-char task IDs in dim. The active/focused node gets its name shown inline. Completed nodes show as unlabeled dots. The log area shows IDs too, with names only on the currently active line.

Plan lifecycle states: ○ draft, ● ready, ◉ editing, ▸ building, ✓ done.

Selecting "Other Tasks" filters the main pane

Pressing Enter on ▸ Other Tasks (3) filters the main pane to show only Other Tasks. The entry stays collapsed — you don't need to see task names in the sidebar because they're right there in the main pane. The ◀ arrow shows what's selected. [a] shows all, [Esc] deselects.

Note the rendering difference: Other Tasks show their subtask dependency tree directly, not the build/review/fix pipeline phases. Nodes are dots — only the active node shows its name. "Fix login redirect" has a 3-subtask chain. "Bump deps" was simple (all dots, done). "Error refactoring" has a fan-out tree with the active node labeled.

Expanding Other Tasks inline (optional)

Pressing [→] or [l] on the Other Tasks entry expands it to show task names inline in the sidebar. The ▾ chevron indicates expanded state. Press [←] or [h] to collapse. This is a preview — the main pane doesn't filter until you press [Enter].

This is useful when you have many Other Tasks and want a quick glance at what's there without switching the main pane away from your current focus.

v1 design decisions

Other Tasks as a single collapsible entry
▸ Other Tasks (3) is one line in the sidebar regardless of how many exist. This keeps the sidebar proportional to plans, not tasks. Three interactions:
- Enter — filter the main pane to Other Tasks (the "list" lives in the main pane)
- → / l — expand inline to preview task names in the sidebar
- ← / h — collapse back to single line

Plan tasks vs Other Tasks — rendering difference
Plan tasks follow the build → review → fix pipeline. Their detail views show three phase sections, and their row views show three phase bars.
Other Tasks are free-form subtask trees with dependencies. They don't have pipeline phases. Their detail views show a raw subtask DAG (no phase labels). Their row views show a single merged progress bar with a completed/total fraction. Some Other Tasks may have review or fix steps visible if those were triggered, but the structure is a dependency graph, not the three-phase pipeline.

DAG nodes are dots, not names
Real subtask names are too long for inline DAG display ("Update webhook signature verification" doesn't fit between connector chars). So:
- All nodes render as state dots: ● done, ◉ active, ○ pending
- Active/focused nodes show their name inline after the dot + short ID
- Completed/pending nodes show only a 4-char task ID in dim
- Log entries show IDs, with name only on the current active line
This keeps the DAG compact and readable at any tree width. Hover/focus on a node to see its full name (future: tooltip or status bar).

Sidebar as scope selector
Every sidebar item (plan or Other Tasks) is a filter. Selecting one narrows the main pane. The ◀ arrow and header text make the active filter obvious. [a] shows all, [Esc] deselects. Plans and Other Tasks are symmetrical — same interaction model.

Plan lifecycle in sidebar
Five states: ○ draft, ◉ editing, ● ready, ▸ building, ✓ done. The [b] build action only appears on ready plans. Drafts get [e] edit + [r] mark ready.

Promote Other Task → plan
[p] create plan on an Other Task generates a plan file and links it. The Other Tasks count decrements; a new plan entry appears in the sidebar. Once promoted, the task gains the build → review → fix pipeline.

Row view column adaptation
When showing all tasks: plan tasks get BUILD/REVIEW/FIX columns, Other Tasks get a merged progress bar spanning those columns. When filtered to Other Tasks only: columns change to TASK / SUBTASKS / TIME — no phase headers at all. When filtered to a plan: standard BUILD/REVIEW/FIX columns (PLAN column hidden since redundant).

Not yet explored:
- Error/failure states and stuck task detail
- Review phase detail (issue cards, code snippets)
- Iteration cycle visualization (build→review→fix loop)
- Drill-down transitions ([n] zoom sequence)
- Tier 3/4 behavior with sidebar (collapse? hide?)
- Other Tasks with review/fix steps visible (hybrid rendering)
