aiki tui — v3 polished screens

Seven screens forming one cohesive experience: task list, task show, build (live + show), review (live + show), fix. Live screens for watching active work. Show screens for examining completed phases. Full-width layout (no sidebar). Consistent chrome: breadcrumb bar, key hints, stage track. Building on v2's tree DAG rendering with refined information hierarchy.
v3 design principles

1. Consistent chrome across all screens
Every screen has three zones: (A) a top bar with breadcrumb path + context, (B) the content area, (C) a bottom bar with contextual key hints. The top bar uses a breadcrumb format: aiki > screen > context. This gives spatial orientation — you always know where you are.

2. Two weights of information
Primary information is white/bright (.wht or .hi). Supporting context is .fg (#777) or .dim (#3a3a44). Status colors (green/yellow/red/cyan) are reserved for state indicators. No element uses more than one status color.

3. Breathing room
v2 was dense. v3 adds strategic whitespace: blank lines between sections, padding around key content blocks. Density is a knob, not a constant — the build/review/fix screens are denser because they're "watching" mode, while task list/show are "reading" mode.

4. No borders on content
v1 used box-drawing for panel borders. v2 used separator lines. v3 drops both — content blocks are separated by whitespace and dim horizontal rules. The only box-drawing chars are for the DAG tree itself. This reduces visual clutter and lets the tree structure be the star.

5. Screen transitions
task list → [Enter] → task show → [b] → build. build completes → auto-advances to review → completes → fix. [Esc] always goes back one level. [q] quits from anywhere.
Screen 1 — Task List

The landing screen. Shows all tasks grouped by status. Each task gets one line with inline status indicators. Tasks with the build/review/fix pipeline show phase progress. Ad-hoc tasks show subtask progress. The list is sorted: in-progress first, then ready, then done.

Key design choice: One line per task, no two-line cards. Task names truncate at ~35 chars. Agent abbreviation and time are right-aligned. Phase progress uses compact B R F letter indicators with color, not full words or bars. This keeps it scannable at a glance while fitting 10+ tasks on screen.

Grouping: Tasks are grouped by status with dim section headers. Within each group, tasks are sorted by priority (p0 first) then recency. The group headers double as a summary: "In Progress (3)" tells you the active workload at a glance.

File-review provenance: Completed pipeline tasks show a compact file-review summary (e.g., 4 files ✓ for all reviewed, or 6 files, 3 reviewed for partial coverage). This surfaces the most valuable provenance signal — review coverage — without drilling in.

Screen 3 — Build

The build screen is "watching" mode — denser, more kinetic, designed for monitoring active work. The tree DAG is the centerpiece. Below it, a live event log streams task completions, parallel starts, and agent activity.

Key difference from task show: The build screen focuses entirely on the active phase. No comments section. No description. Just the tree, a compact header, and the event log. It's what you see when you run aiki build webhooks.md or drill from task show.

The event log uses the same format as v2 but with slightly better density: parallel events on one line with │ separator, timestamps left-aligned, agent abbreviations color-coded. The log auto-scrolls but you can press [Space] to pause and scroll manually.

Scene 3e — Build Show (retrospective)

aiki build show — examining a completed build after the fact. This is "reading" mode, not "watching" mode. All nodes are resolved (green or red). Every subtask gets its full name, timing, and agent. The tree renders the same DAG as during live build, but with full annotations.

Key difference from live build: During live build, only active nodes get names (to save space for the tree structure). In build show, every node gets a name and timing because the tree is static — no animation, no event log, no urgency. You're reviewing the record.

Subtask list below the tree: The tree gives the shape (parallelism, dependencies). Below it, a flat list gives the details: full subtask name, agent, duration, status. This dual view — tree for structure, list for detail — avoids cramming too much text into the DAG.

Screen 4 — Review

The review screen shows the completed build tree (dimmed) above the active review work. Review tasks are typically simpler than build — 1-3 parallel review passes (code review, test verification, security scan). Each review task can produce issues.

Issue surfacing: When a review task finds issues, they appear inline below the review tree as a compact list: severity indicator + one-line description. These issues become the input for the fix phase. The issue count appears in the stage track.

Visual continuity: The build tree stays visible (green, dimmed) above the review section. This creates a sense of progression — you can see where you've been.

Scene 4d — Review Show (retrospective)

aiki review show — examining a completed review after the fact. This is the audit view: what was reviewed, who reviewed it, what issues were found, and what files were flagged. Unlike the live review screen, this has no event log or progress indicators — everything is resolved.

Three sections:
1. Review passes — what checks ran, who ran them, how long each took
2. Issues — full detail on each issue: severity, source, description, affected files
3. Files touched — which files the build modified, cross-referenced with which review pass examined them. This is the provenance chain: build produced these files, review checked them, issues were found in specific files.

This screen drives the fix phase. When you run aiki fix, it reads from the review results. The issue list here is exactly what becomes fix tasks. File references help the fix agent know where to look.

Screen 5 — Fix

The fix screen shows fix tasks derived from review issues. Each fix task is linked to its source issue. Fix tasks can run in parallel (independent issues) or sequentially (if one fix depends on another). The tree DAG renders fix tasks just like build subtasks.

Issue → fix linking: Each fix task shows which issue it addresses via a dim reference (← issue 1). When a fix completes, its corresponding issue is annotated with fixed. This creates a clear audit trail: issue found → fix applied.

The re-review gate: After all fixes are applied, aiki can optionally run a targeted re-review (just the changed files, not the full review). If the re-review passes, we're done. If not, another fix iteration starts. The iteration counter tracks this.

v3 design notes & rationale

What changed from v2:
- Breadcrumb navigation replaces the title bar context. aiki > tasks > id > build gives constant spatial orientation. You always know where you are in the hierarchy.
- Consistent chrome: every screen has the same three zones (breadcrumb, content, key hints). v2 had inconsistent layouts between scenes.
- Breathing room: strategic blank lines and separator rules. v2 was dense everywhere. v3 varies density by context — reading modes are spacious, watching modes are dense.
- No panel borders: content is separated by whitespace and dim ─── rules, not ╭╮╰╯│ boxes. The DAG tree structure is the only use of box-drawing.
- Stage track compact form: B:✓ R:2 issues F:1/3 instead of full words on the task list. Full form on detail screens where space allows.
- Issue tracking in review/fix: v2 showed issues as a count. v3 shows them as a persistent list that flows through review → fix, with status tracking per issue.

Live vs retrospective (the "show" distinction):
Each phase has two modes:
- Live (aiki build, aiki review, aiki fix) — watching mode. Dense, kinetic. Event log streaming. Active nodes with names. Progress indicators.
- Retrospective (aiki build show, aiki review show) — reading mode. Static, spacious. Full subtask list with names/timing/agents. File matrix for review. No event log — everything is resolved.
The live screen is what you see during execution. The show screen is what you examine after. Same data, different information needs. Live answers "what's happening now?" Show answers "what happened and what do I do next?"

Screen hierarchy:
task list → task show → build/review/fix (live) → subtask drill-down
task show → build show/review show (for completed phases)
Navigation: [Enter] goes deeper. [Esc] goes back. [q] quits. Auto-advance between phases in live mode. In task show, clicking a completed phase opens its "show" retrospective view.

What's NOT in v3 (future work):
- Tier 2+ scaling (row view, group view, fleet) — covered in v1 scene 8, needs v3 treatment
- Multi-task dashboard (multiple tasks at different pipeline stages simultaneously)
- Plan editor integration
- Keyboard shortcuts for task actions (close, comment, reassign)
- Color theme customization / --ascii mode
- Log scrollback / search
