v2 — tree DAG + stage track

Combines v1's tree DAG rendering (box-drawing connectors: ━┬━ ├━ ╰━ │) with v2's full-screen layout and stage track header. The tree naturally shows lane structure through branching — parallel branches are parallel lanes. No need for lane IDs, lane rows, or lane names.

How it works:
- The subtask DAG renders as a left-to-right tree using ━┬━ for fan-out, ├━/╰━ for branches, │ for vertical continuation
- Nodes are dots: ● done (green), ◉ active (yellow bold), ○ pending (dim), ✗ failed (red)
- Only the active node (◉) gets a truncated task name inline
- Agent abbreviation sits at the right edge of active branches
- The stage track (▸ build ○ review ○ fix) is always visible in the header
- Each stage (build, review, fix) has its own tree
- Completed stages keep their tree visible (green dots, dim connectors)

Key insight: The tree shape is the lane visualization. A fan-out from ━┬━ creates parallel branches — those branches are lanes. Fan-in with ━╯ reconverges. No extra abstraction needed.

v2 design decisions

Tree DAG rendering (from v1)
The subtask DAG renders as a left-to-right tree using box-drawing characters. This is the same rendering approach as v1's detail pane, but now full-screen with no sidebar. Connectors: ━┬━ fan-out, ├━/╰━ branches, │ vertical, ━╯ fan-in. Nodes: ● done (green), ◉ active (yellow bold), ○ pending (dim), ✗ failed (red).

Lanes are implicit in tree shape
A fan-out (━┬━) creates parallel branches — those branches are lanes. No need for lane IDs, lane rows, or lane labels. The tree shape communicates everything: how many things run in parallel, what depends on what, where bottlenecks are. When two branches converge (━╯), that's a fan-in dependency.

Stage track header (from v2)
Always visible: ▸ build 3/6 ○ review ○ fix. Three stages left-to-right with status indicators:
- ▸ build 4/6 — active stage, yellow, task progress
- ✓ build 10/10 — completed, green
- ○ review — pending, dim
- ✓ review 2 issues — done with issues (orange)
- ✓ review clean — done, no issues

Vertical layout: three stage trees
The screen shows up to three trees vertically, one per stage. The active stage's tree has yellow/green active nodes. Completed stages keep their green trees visible as context (with dim connectors). Each stage header shows its name and total time.

Only active nodes get names
Task names are 5–8 words (30–45 chars) — too long for every node. Only ◉ (active) and ✗ (failed) nodes show truncated names inline (first 2–3 words + …). Completed nodes are just ●. Pending nodes are ○. Agent abbreviation sits at the right edge of active branches.

Drill-down
[n] on any branch zooms into session detail: full task names, completion summaries, live agent output. [b] returns to tree overview. The stage track stays visible in drill-down mode.

Not yet explored:
- Iteration visualization (iter 2+ re-review cycle)
- How the tree collapses when build has many nodes (10+ nodes, tree gets wide)
- Multi-agent coordination: how to show which agent owns which branch
- Tier 2+ scaling for this tree approach (many trees → row compression?)
- Review issue detail in drill-down
- Fix tasks that trigger their own review (recursive pipeline)
- Navigation: cursor movement within the tree (which node is "focused"?)
