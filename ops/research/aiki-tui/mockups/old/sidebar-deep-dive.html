CWD in the sidebar

The original Option A had aiki status ops/now in the title bar, but the sidebar header just said "plans". The sidebar IS the directory — its header should say so. The plan filenames are then relative paths within that dir. This makes the sidebar feel like a scoped file-tree, not a generic list.

Planless tasks

Not every task comes from a plan file. aiki task start "Fix login redirect" creates a task with no plan backing it. These need a home in the sidebar. Three options explored below:

Option 1: A ── tasks ── divider below plans, listing planless tasks by name
Option 2: A virtual (no plan) entry that acts like a plan but groups all loose tasks
Option 3: Planless tasks only appear in the main pane; sidebar shows a count badge

Draft vs ready plans

Not all plan files are ready to build. A plan goes through a lifecycle:

○ draft (dim) — file exists, still being written. Can't build yet.
● ready (cyan) — plan is complete, waiting for someone to run aiki build.
◉ editing (magenta, blink) — someone has it open in an active session right now.
▸ building (yellow) — aiki build is running, tasks in progress.
✓ done (green) — all tasks complete.

The key UX consequence: [b] build is only available on ready plans. You can't accidentally kick off a build from a half-written draft. Drafts show [e] edit and [r] mark ready instead.

Ready plan = buildable

When a ready plan is selected, the main pane shows a summary and the [b] build action is prominent. This is the "green light" moment — the plan is complete, reviewed, and waiting to be executed. The preview shows what will happen: task breakdown, estimated scope.

The [e] edit action is still available (you can always go back to draft). And [b] build transitions the plan to "building" and kicks off task creation.

Plan lifecycle — visual reference

  ○ draft      Plan file exists, still being written
                 Actions: [e] edit  [r] mark ready

  ◉ editing    Someone has it open in an active session
                 Actions: [f] attach to session

  ● ready      Plan is complete, waiting to be built
                 Actions: [b] build  [e] edit

  ▸ building   aiki build running, tasks in progress
                 Actions: [n] drill  [s] stop  [Enter] focus

  ✓ done       All tasks complete
                 Actions: [n] drill  (inspect completed work)

Transitions:
draft → ready — user runs [r] mark ready or aiki plan ready <file>
ready → building — user runs [b] build or aiki build <file>
building → done — all tasks complete (automatic)
ready → draft — user runs [e] edit (plan re-opens for changes)
done → building — user re-runs build (new iteration)

How does a plan become "ready"?
Explicit: aiki plan ready payments.md or frontmatter status: ready
The plan file itself defines what "complete" means — aiki doesn't enforce structure. It's a human decision: "I'm done writing this, it's ready to build."

What about plans that skip the draft stage?
If you write a plan in one shot and immediately mark it ready, that's fine. aiki plan add --ready "Quick fix for login bug" creates a ready plan directly. The draft state exists for plans that need iteration, not as a mandatory gate.
Design decisions and open questions

CWD in sidebar
The sidebar header IS the cwd. ╭─ ops/now/ ────────╮ tells you the scope immediately. At repo root (Scene 8), it shows the repo name with subdirectories as nested groups. Title bar is just aiki status — no redundancy.

Planless tasks: "tasks" section wins
Scene 2a (explicit task list with ── tasks ── divider) is the strongest option because:
- Tasks are named, so they're navigable (unlike a count badge)
- They live in the same j/k navigation flow as plans
- The divider creates a clear visual break — you know what's planned vs ad-hoc
- Degrades gracefully to "all tasks" when no plans exist (Scene 7)

The virtual (no plan) entry (Scene 2b) is too abstract — it's a group pretending to be a single entry. A count badge hides the tasks and requires switching panes to see them.

"no plan" label in main pane
When showing all tasks in compressed view (Scene 4), planless tasks show no plan in dim on the task header line. In the row view (Scene 5), the PLAN column shows —. Both are subtle enough not to nag — ad-hoc tasks are fine, not an error state.

Plan filtering
Selecting a plan in the sidebar filters the main pane (Scene 6). The ◀ arrow and showing payments.md — 3 tasks header make the filter state obvious. [a] to show all, [Esc] to deselect. Selecting a task in the "tasks" section filters to that single task's pipeline view.

[p] create plan
When a planless task is focused, the hotbar shows [p] create plan — promotes ad-hoc work into planned work. Opens $EDITOR with a plan template pre-filled from the task description. This makes the plan/planless distinction feel like a natural workflow progression, not a rigid categorization.

Plan lifecycle in sidebar (Scenes 10–15)
Five distinct states: ○ draft, ◉ editing, ● ready, ▸ building, ✓ done. The key UX gate: [b] build is only available on ready plans. Draft plans show [e] edit + [r] mark ready. This prevents accidentally building from half-written plans.

The draft/ready distinction replaces the old "idle" state. Previously, a plan with no icon could be either incomplete or ready-but-unbuilt — ambiguous. Now the intent is explicit.

Open questions:
- Should completed plans (all tasks done) auto-collapse or stay visible?
- Should the sidebar show task counts per plan? e.g., payments.md ▸ 3
- At Tier 3/4 scale, should the sidebar collapse to just plans (hiding individual tasks)?
- How should the sidebar handle nested directories? (Scene 8 hints at this with subdirectory groups)
- Should [e] edit on a ready plan transition it back to draft, or stay ready?
- Can agents mark plans ready, or is it always a human decision?
