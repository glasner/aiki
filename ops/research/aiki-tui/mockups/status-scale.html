aiki status at scale

Progressive density — 1 task to 100+ agents
Four density tiers — auto-selected by task count:

1–3 tasks → Full DAG view with parallel streams, subtask labels, inline issues
4–15 tasks → One row per task, phase bars instead of DAGs, issue counts
16–50 tasks → Grouped by status, sparkline bars, only outliers named
50+ tasks → Fleet dashboard — histogram, counts, only stuck/failed tasks listed

User can always [number] to zoom into a single task at full DAG detail, or --density=full|row|group|fleet to override.

Progressive density model:

Tier 1 (1–3 tasks): Full DAG with parallel streams, subtask labels, inline issue details. You care about every subtask.

Tier 2 (4–15 tasks): One row per task, phase shown as a bar (████░░░░) not a DAG. Parallel work collapsed to bar fill rate. Issue counts inline. You care about which tasks are progressing vs stuck. Key addition: [/] filter and --stuck to slice the list.

Tier 3 (16–50 tasks): Grouped by status (building/reviewing/fixing/done/stuck). Per-agent breakdown with mini stacked bars. Only stuck tasks shown individually. You care about fleet throughput and bottlenecks.

Tier 4 (50+ tasks): Fleet dashboard — histogram bars for pipeline stages, agent table with rates, throughput sparkline, only stuck/failed listed individually. You care about system health, not individual tasks. Agent-focused sub-view shows per-instance health and common failure patterns.

Drill-down always available: Any tier can zoom to any single task at Tier 1 detail via [n] focus. You go fleet → agent → task → subtask DAG. And [b] back zooms out.

Auto-selection logic:
aiki status picks tier based on active task count. Override with:
• aiki status --density=full (force Tier 1 for everything — may be long)
• aiki status --density=row (force Tier 2)
• aiki status --density=group (force Tier 3)
• aiki status --density=fleet (force Tier 4)
• aiki status --stuck (filter to problems only at any tier)
• aiki status --agent cursor (filter to one agent)

Key insight: The information you need changes with scale. At 3 tasks you want "what subtask is running." At 100 tasks you want "what's the throughput and what's stuck." The transition from bar to sparkline to histogram is the same visual vocabulary at different densities — ████░░ works whether it's one task's build phase or an entire agent's completion rate.
