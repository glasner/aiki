## Executive Summary

Two commands form a complete development workflow:

- **`aiki spec <path-or-text>`** - Interactive spec authoring session with Claude
- **`aiki build <spec-path>`** - Autonomous: creates plan task with subtasks, then executes

**The Full Workflow:**

```
aiki spec "add auth"   →  Spec document (ops/now/add-auth.md)
aiki build             →  Creates plan task + build task, executes all subtasks
```

**Terminology:**

| Term | What it is | Created by |
|------|-----------|------------|
| **Spec** | Document (human-readable requirements) | `aiki spec` command |
| **Plan** | Task with subtasks (machine-readable breakdown) | `aiki build` (internal step) |
| **Build** | Orchestration task | `aiki build` command |

**Key insight:** Developers write the high-level plan in their spec document. The "plan task" is just a machine-readable breakdown for agent execution — not something humans need to inspect or customize separately.

**Benefits:**
- Two commands instead of three (simpler mental model)
- Spec is the human artifact, plan is the machine artifact
- "Plan" refers to the task created by build (not a separate command)
- Can still inspect plan before execution with `--plan-only`

**Pipeable:**
```bash
# Full pipeline
aiki build ops/now/feature.md | aiki review | aiki fix

# Step by step
aiki spec "Add dark mode"              # Interactive spec authoring
aiki build ops/now/add-dark-mode.md    # Execute implementation
```

## Commands

### `aiki spec`

Interactive spec authoring session with Claude.

```bash
aiki spec <path-or-text...> [options]
```


**Arguments:**
- `<path-or-text...>` - Variadic argument (one or more words). Three modes:
  - Existing `.md` file path → **edit mode** (refine existing spec)
  - Non-existent `.md` file path → **create at path mode** (new spec at specified location)
  - Multiple words or text → **autogen mode** (create with autogenerated filename)

**Mode detection:**
1. If first arg ends with `.md` and file exists → edit mode
2. If first arg ends with `.md` and file doesn't exist → create at path mode (parse initial idea from filename)
3. Otherwise, join all args with spaces → autogen mode
4. Quotes are optional and supported for backward compatibility

```bash
# Interactive - prompted for multi-line input
aiki spec
> Describe what you want to spec (Shift+Enter for newlines):
> Add a spec command that...
> - takes file path or text
> - runs interactively
> [Enter to submit]

# Edit existing file
aiki spec ops/now/feature.md

# Create at specific path (file doesn't exist - idea parsed from filename)
aiki spec ops/now/dark-mode.md        # Initial idea: "dark mode"
aiki spec ops/now/user-auth-v2.md     # Initial idea: "user auth v2"

# Autogen filename - no quotes needed (variadic args)
aiki spec Add user authentication
aiki spec Fix the login bug

# Quotes still work (backward compatible)
aiki spec "Add user authentication"

```

**Options:**
- `--template <name>` - Spec template (default: `aiki/spec`)
- `--agent <name>` - Agent for spec session (default: `claude-code`)

**Note:** No `--async` flag. Unlike `build`/`review`/`fix` which support autonomous execution, `spec` is always interactive and user-driven.

**Output:**
```xml
<aiki_spec cmd="spec" status="ok">
  <completed spec_task="xtuttnyvykpulsxzqnznsxylrzkkqssy" file="ops/now/add-user-auth.md">
    Spec session completed.

    Created: ops/now/add-user-auth.md
    Sections: Vision, Requirements, Open Questions
  </completed>
</aiki_spec>
```

### `aiki build`

Creates a plan task (implementation breakdown) and orchestrates execution of all subtasks.

```bash
aiki build <spec-path> [options]
```

**Arguments:**
- `<spec-path>` - Path to spec file (e.g., `ops/now/my-feature.md`)

**Options:**
- `--async` - Run asynchronously, return build task ID immediately
- `--plan-only` - Create plan task only (don't execute subtasks)
- `--template <name>` - Build task template (default: `aiki/build`)
- `--agent <name>` - Agent for build orchestration (default: `claude-code`)

Note: No `--start` flag — build tasks run to completion autonomously.

**Behavior:**
1. Create build task from `aiki/build` template with `data.spec=<spec-path>`
2. Build agent reads spec file
3. Build agent creates plan task with subtasks (the implementation breakdown)
4. Build agent executes each subtask in order
5. Output build task ID to stdout

**With `--plan-only`:**
1. Create build task, agent creates plan task with subtasks
2. Stop before executing subtasks
3. Output plan task ID to stdout

**Output (stdout when piped):**
```
build9012
```

**Output (stderr):**
```xml
<aiki_build cmd="build" status="ok">
  <completed build_task="build9012" plan_task="plan5678" subtasks="5" duration_ms="120000">
    Build completed successfully.

    Plan: plan5678
    1. Add JWT library dependency (done)
    2. Create auth middleware (done)
    3. Add login endpoint (done)
    4. Add token validation (done)
    5. Write tests (done)
  </completed>
</aiki_build>
```

---

### `aiki build show`

Shows the build task and plan task for a spec file.

```bash
aiki build show <spec-path>
```

**Arguments:**
- `<spec-path>` - Path to spec file (e.g., `ops/now/my-feature.md`)

**Behavior:**
1. Find build task with `data.spec=<spec-path>` (most recent if multiple)
2. Find associated plan task
3. Display both tasks with subtask status

**Output:**
```
Build Task: build-xyz (orchestrator)
  Status: in_progress
  Started: 2 min ago

Plan Task: plan-abc
  Source: ops/now/add-auth.md
  Progress: 3/5 subtasks

  ├── [✓] Add database schema
  ├── [✓] Create API endpoints
  ├── [✓] Add middleware
  ├── [ ] Write tests
  └── [ ] Update documentation
```

**Error handling:**
- If no build task found: Error with suggestion to run `aiki build <spec-path>` first
- If multiple builds: Show the most recent one (with note about how many exist)

## Use Cases

### 1. Full Pipeline - Start to Finish

```bash
# Step 1: Create spec interactively
aiki spec "Add dark mode support"
# ... collaborate with agent to write spec ..
# Creates: ops/now/add-dark-mode-support.md

# Step 2: Build from spec
aiki build ops/now/add-dark-mode-support.md | aiki review | aiki fix
```

### 2. Inspect Plan Before Building

```bash
# Create spec
aiki spec ops/now/risky-refactor.md

# Create plan only (don't execute)
aiki build ops/now/risky-refactor.md --plan-only

# Inspect the plan
aiki build show ops/now/risky-refactor.md

# Edit plan if needed
aiki task show <plan-task-id>
aiki task add --parent <plan-task-id> "Add rollback mechanism"

# Now execute (creates new build that uses existing plan)
aiki build ops/now/risky-refactor.md
```

### 3. Async Build

```bash
# Start build in background
aiki build ops/now/complex-feature.md --async
# Returns: build1234

# Check build status
aiki build show ops/now/complex-feature.md
# Or: aiki task show build1234
```

### 4. Refine Existing Spec

```bash
# Edit an existing spec file
aiki spec ops/now/feature.md
# ... collaborate with agent to refine the spec ...
```

## Task Templates

### Spec Template (`aiki/spec`)

**Location:** `.aiki/templates/aiki/spec.md`

**Template Variables:**
- `data.spec_path` - Absolute path to the spec file being edited/created
- `data.is_new` - Boolean, `true` if creating a new spec, `false` if editing existing
- `data.initial_idea` - String, initial idea parsed from filename or description

```markdown
---
version: 1.0.0
type: spec
assignee: claude-code
interactive: true
---

# Spec: {data.spec_path}

Guide the user through creating or refining a spec document.

# Instructions

You are helping the user write a spec at `{data.spec_path}`.

**For new specs:** Work through the subtasks in order (they are defined at the end of this template).

**For existing specs:**
1. Read and summarize current state
2. Identify which phase the spec is in (incomplete sections? unresolved questions?)
3. Start the appropriate subtask for where work is needed
4. Work through remaining subtasks as needed

**Throughout:**
- Write to the spec file incrementally (don't wait until the end)
- Keep the conversation focused
- Use subtasks to track progress through the spec workflow
- Track open questions as subtasks under "Resolve open questions", NOT in the spec file's "Open Questions" section

## Spec Structure (suggested)

Use this structure for new specs:

    # <Title>

    **Date**: <today>
    **Status**: Draft
    **Purpose**: <one line>

    **Related Documents**:
    - [Related Doc](path/to/doc.md) - Brief description

    ---

    ## Executive Summary

    <2-3 sentences describing what this is and why>

    ---

    ## User Experience

    <If applicable - command syntax and options>

    ---

    ## How It Works

    <Explanation of the mechanism/flow>

    ---

    ## Use Cases

    <Concrete examples of how this is used>

    ---

    ## Implementation Plan

    <Phases and deliverables>

    ---

    ## Error Handling

    <If applicable - error scenarios and handling>

    ---

    ## Open Questions

    1. ...
    2. ...

    ---


# Subtasks

## Clarify user intent and requirements

1. Ask clarifying questions where you have them:
   - What problem does this solve?
   - Who is the user/audience?
   - What are the key requirements?
   - Any constraints or non-goals?
3. Close subtask when you have enough clarity to draft the spec

## Draft initial spec structure

1. Create the spec file using the template structure
2. Fill out Summary, Requirements, and Open Questions sections
3. Write to the file incrementally as you work
4. Close subtask when initial draft is complete

## Resolve open questions

1. Work through each question subtask with the user
2. Update spec based on answers
3. Close each question subtask when resolved
4. Add any new questions that emerge as additional subtasks
5. Close this subtask when all question subtasks are resolved

## Validate completeness

1. Review the complete spec with user
2. Confirm all sections are filled out
3. Verify open questions are addressed or documented
4. Get user confirmation that spec is ready
5. Close subtask and parent task when validated

**Spec completeness criteria:**
- All required sections have content (no TODOs or placeholder text)
- Open Questions section is either resolved or documented for later
- User explicitly confirms "this spec is complete" or "ready to implement"
```

### Build Template (`aiki/build`)

**Location:** `.aiki/templates/aiki/build.md`

```markdown
---
version: 1.0.0
type: build
assignee: claude-code
---

# Build: {data.spec}

**Overall Goal**: Read the spec, create a plan task with implementation subtasks, then execute each subtask.

# Subtasks

## Read and understand the spec

Read the spec file at `{data.spec}` and identify the implementation steps needed.

Examine the spec to understand:
- What is the goal/vision?
- What are the requirements?
- What are the constraints?
- Are there open questions that need resolving?

Summarize the implementation steps you'll need to create. Each step should have a coherent, testable change.

## Create plan task

Create the plan task (a container for subtasks):

```bash
aiki task add "Plan: <spec title from the spec file>" \
  --source task:{task.id} \
  --source file:{data.spec}
```

Save the returned task ID - you'll use it as the parent for subtasks.

Then add metadata to link it to the spec:
```bash
aiki task set-data <plan_task_id> spec {data.spec}
```

## Add implementation subtasks

For each implementation step identified, create a subtask under the plan task:

```bash
aiki task add "<step description>" --parent <plan_task_id>
```

Guidelines for subtasks:
- Each subtask should be a discrete, actionable step
- Include enough context in names for the executing agent
- Order subtasks logically (dependencies first)
- Mark blocking relationships between subtasks when one must complete before another can start

After creating subtasks, set up blocking relationships:
```bash
# If subtask 2 depends on subtask 1 completing first:
aiki task update <plan_task_id>.2 --blocked-by <plan_task_id>.1
```

This enables parallel execution of independent subtasks while ensuring dependencies are respected.

## Execute subtasks

Execute each subtask of the plan task sequentially:

```bash
aiki task run <plan_task_id>.1
aiki task run <plan_task_id>.2
aiki task run <plan_task_id>.3
...
```

Run them in order. If a subtask fails, stop and report the failure - don't continue.

As you complete each subtask, add a progress comment:
```bash
aiki task comment {task.id} "Completed subtask N of M"
```

## Close build task

When all plan subtasks are complete, close this build task:

```bash
aiki task close --comment "Build completed: all N subtasks done. Plan task: <plan_task_id>"
```
```

**Note**: The build task creates and orchestrates the plan task. This separation allows:
- Tracking build progress independently
- Re-running builds (creates new plan)
- Inspecting plan before execution with `--plan-only`

## Error Handling

### Spec Command

| Scenario | Behavior |
|----------|----------|
| Path exists + ends in `.md` | Edit mode - refine existing spec |
| Path doesn't exist + ends in `.md` | Create at path mode - parse idea from filename |
| Path doesn't exist + no `.md` | Autogen mode - treat as description text, generate filename |
| Path exists but not `.md` | Error: "Not a markdown file" |
| Path outside repo | Error: "Path must be inside repository" |
| Permission denied | Error: "Cannot write to <path>" |
| Output dir doesn't exist | Create directory (like `mkdir -p`) |
| File locked/open | Error: "File is locked, close other sessions first" |
| Multiple concurrent sessions | Allow - creates separate task for each session |
| Git in bad state (mid-rebase, etc.) | Allow spec creation (read-only git status) |

### Build Command

| Scenario | Behavior |
|----------|----------|
| Spec file doesn't exist | Error: "Spec file not found: <path>" |
| Invalid spec file (not .md) | Error: "Spec file must be markdown (.md)" |
| Plan task creation fails | Stop, return error with agent output |
| Subtask execution fails | Stop build, return failed subtask |
| Async build timeout | Return task ID, let it run in background |

## Open Questions

### Spec Command

1. ~~**Filename generation**: How to handle collisions?~~
   - Decision: Increment suffix (`-2`, `-3`, etc.). No prompt - just pick the next available name.

2. ~~**Session resumption**: If user quits mid-spec, can they resume?~~
   - Decision: Yes. `aiki spec <file>` checks for existing open `type: spec` task.

3. ~~**Default output directory**: Should it always be `ops/now/` or configurable?~~
   - Decision: Hardcoded to `ops/now/` for v1. Can revisit in v2.

4. ~~**Non-interactive mode**: Should `aiki spec` support `--async` for agents creating specs?~~
   - Decision: No. Spec is always user-driven and interactive.

5. ~~**Stdin support**: Should v1 support stdin for automation?~~
   - Decision: No, push to v2. Keep v1 interactive-only.

6. ~~**Concurrent sessions**: What if user runs `aiki spec file.md` in two terminals?~~
   - Decision: Allow - creates separate task for each session.

### Build Command

1. ~~**Build without existing plan** - Should `aiki build` detect existing plan tasks?~~
   - Decision: Yes. Check for existing plan task with `data.spec=<spec-path>` before creating new one.

2. ~~**Incremental builds** - Should `aiki build` resume from partial plan?~~
   - Decision: Future enhancement. For v1, always start fresh.

## Summary

**Two commands, one workflow:**

| Command | Mode | Input | Does | Output |
|---------|------|-------|------|--------|
| `aiki spec` | Interactive | Description or path | Collaborative spec authoring | Spec file |
| `aiki build` | Autonomous | Spec file | Creates plan task + executes | Build task ID |
| `aiki build show` | Query | Spec file | Shows build + plan status | Task details |
| `aiki build --plan-only` | Autonomous | Spec file | Creates plan task only | Plan task ID |

**The full pipeline:**

```bash
# 1. Create spec interactively
aiki spec "Add user authentication"

# 2. Execute implementation
aiki build ops/now/add-user-authentication.md

# OR: With review
aiki build ops/now/add-user-authentication.md | aiki review | aiki fix

# OR: Inspect plan first
aiki build ops/now/add-user-authentication.md --plan-only
aiki build show ops/now/add-user-authentication.md
aiki build ops/now/add-user-authentication.md
```

**Key design principles:**

1. **Spec is collaborative** - User and agent work together to define requirements
2. **Build is autonomous** - Creates plan and executes without intervention
3. **Plan is internal** - Machine-readable task breakdown, not a separate command
4. **Tasks are composable** - Each phase creates tasks that can be inspected or re-run
5. **Clear terminology** - Spec (document), Plan (task), Build (orchestrator)
