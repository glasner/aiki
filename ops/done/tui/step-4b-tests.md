## Problem

The existing `tui_snapshot_tests.rs` tests the old `Chat` → `pipeline_chat` rendering path. After the rewrite, these tests are invalid. New tests should follow the ratatui testing pyramid and validate each flow against the screen state mockups.

---

## Testing Pyramid

From ratatui best practices — each layer is faster and more isolated than the one below:

```
         ┌──────────┐
         │   PTY    │  (optional — only for terminal-specific behavior)
        ┌┴──────────┴┐
        │ Integration │  Full UI via TestBackend + insta snapshots
       ┌┴────────────┴┐
       │ Widget/Buffer │  Render to Buffer, assert on cells
      ┌┴──────────────┴┐
      │  State/Logic    │  Pure update() + view() — no terminal
      └────────────────┘
```

**Guideline:** State tests should be the majority and the fastest. Widget/buffer tests cover rendering correctness. Snapshot tests are the safety net for layout regressions.

## Layer 3: Buffer-Level Render Tests (in `render.rs`)

Render `Vec<Line>` into a `Buffer`, assert on cells directly. Tests styling and layout.

```rust
#[test]
fn phase_header_starts_with_gou_symbol() {
    let lines = components::phase(0, "plan", Some("claude"), vec![]);
    let area = Rect::new(0, 0, 80, 5);
    let mut buf = Buffer::empty(area);
    render_lines(&lines, &mut buf, area, &Theme::dark());
    assert_eq!(buf[(0, 0)].symbol(), "合");
}

#[test]
fn child_done_has_green_check() {
    let lines = components::phase(0, "plan", None, vec![
        ChildLine { text: "path.md".into(), style: ChildStyle::Done, meta: None },
    ]);
    let area = Rect::new(0, 0, 80, 5);
    let mut buf = Buffer::empty(area);
    render_lines(&lines, &mut buf, area, &Theme::dark());
    // Check that ✓ appears on line 1 with green color
    assert_eq!(buf[(2, 1)].symbol(), "✓");
    assert_eq!(buf[(2, 1)].fg, Color::Green);
}

#[test]
fn render_respects_area_bounds() {
    // 20 lines of content, only 5 rows of space — should not panic
    let lines: Vec<Line> = (0..20).map(|i| Line { text: format!("line {i}"), .. }).collect();
    let area = Rect::new(0, 0, 80, 5);
    let mut buf = Buffer::empty(area);
    render_lines(&lines, &mut buf, area, &Theme::dark());
}

#[test]
fn meta_is_right_aligned() {
    let lines = components::phase(0, "task", Some("claude"), vec![
        ChildLine { text: "heartbeat".into(), style: ChildStyle::Active, meta: Some("12s".into()) },
    ]);
    let area = Rect::new(0, 0, 80, 5);
    let mut buf = Buffer::empty(area);
    render_lines(&lines, &mut buf, area, &Theme::dark());
    // "12s" should be right-aligned: starts at column 77
    assert_eq!(buf[(77, 1)].symbol(), "1");
}
```

**Note:** insta snapshots capture text only, not colors. Buffer-level tests are the only way to verify styling (colors, bold, dim).

## Layer 4: Snapshot Tests (in `tui_snapshot_tests.rs`)

Full flow snapshots using `TestBackend` + `insta`. Each test constructs a Model with a specific state, calls `view()`, renders via `TestBackend`, and snapshots.

**Consistent terminal size:** 80 columns, 40 rows for all snapshots.

### Per flow

**Flow 1: task run (6 tests)**
- `test_task_run_creating_workspace` — loading state
- `test_task_run_active` — heartbeat + elapsed
- `test_task_run_done` — summary + next action
- `test_task_run_failed` — error
- `test_task_run_parent_mid_flight` — subtask table with mixed status
- `test_task_run_parent_done` — all subtasks done

**Flow 2: build (7 tests)**
- `test_build_validating` — plan loading
- `test_build_decompose_active` — plan done + decompose with placeholder table
- `test_build_subtasks_arriving` — decompose active + partial subtask table
- `test_build_subtasks_populated` — decompose done + full table
- `test_build_mid_flight` — ✓/▸/◌/○ in table + active lanes
- `test_build_subtask_failed` — ✗ in table + error in lane
- `test_build_all_done` — all ✓ + summary

**Flow 2 continued: build -f (4 tests, graph-driven — same `build::view()` with fix tasks in graph)**
- `test_build_fix_review_approved` — build + review ✓ + summary
- `test_build_fix_iteration_2` — review issues + Iteration 2 + fix
- `test_build_fix_regression_approved` — full cycle
- `test_build_fix_max_iterations` — warning message

**Flows 4-7: review (4 tests)**
- `test_review_plan_issues` — plan path + issues
- `test_review_code_approved` — path --code + approved
- `test_review_task_active` — [id] name + heartbeat
- `test_review_session_issues` — N completed tasks + grouped issues

**Flow 8: fix (3 tests)**
- `test_fix_no_actionable` — approved shortcut
- `test_fix_single_iteration` — fix → decompose → loop → review
- `test_fix_followup_active` — mid-fix with active lanes

**Progressive dimming (2 tests)**
- `test_dimming_active_phase` — earlier phases dimmed
- `test_dimming_all_done` — everything dimmed except summary

**Width-variant tests (3 tests)**

Render the same build mid-flight state at different widths to verify truncation, right-alignment, and no panics:

- `test_width_40` — narrow (truncation, meta overflow)
- `test_width_80` — standard
- `test_width_120` — wide (no misalignment)

Each uses the same Model data but different terminal width. Snapshot captures layout at each width.

### Snapshot workflow

```bash
# Run tests — new/changed snapshots are "pending"
cargo test

# Review pending snapshots interactively
cargo insta review

# Accept all (use with caution)
cargo insta accept
```
