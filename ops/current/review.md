Findings**

- `cli/src/commands/acp.rs:1255` builds `original_text` by concatenating *all* `type:"text"` elements from the incoming prompt, but the rewrite logic at `cli/src/commands/acp.rs:1298-1312` only replaces the first `type:"text"` entry and leaves the rest untouched. Whenever the IDE sends more than one text chunk (the design doc explicitly calls this out as a challenge in `ops/current/event-dispatch-gap-analysis.md:708-713`), the user’s original text still sits in the later entries, so the agent will see `modified_prompt + original_prompt` concatenated together. That makes Phase‑1’s “flows can rewrite the entire prompt” goal unattainable unless the IDE happens to use a single text chunk. The fix is to either clear the remaining text items or rebuild the prompt array with a single synthesized text entry while preserving non-text resources.

- `prompt_requests` gets populated for every `session/prompt` in `cli/src/commands/acp.rs:644-651`, but entries are removed only when the response contains `stopReason == "end_turn"` (`cli/src/commands/acp.rs:724-765`). Any JSON-RPC error or non‑`end_turn` stop reason leaves the entry behind forever, so long sessions that hit token limits/refusals steadily leak `Arc<str>` keys. Beyond the unbounded growth, it also means a future JSON-RPC response that happens to reuse an old ID (allowed by spec) will match the stale session_id instead of the real one, preventing the PostResponse event from firing for that turn. Make sure to drop the mapping whenever *any* response arrives for a tracked request (success, failure, or explicit error).

- The Phase‑1 requirement was “Forward original response to IDE, then send autoreply to agent” (`ops/current/event-dispatch-gap-analysis.md:734-745`), but the code enqueues the autoreply before the agent→IDE thread prints the original response. `handle_post_response()` queues work on the channel in `cli/src/commands/acp.rs:1390-1440` while the surrounding loop only `println!`s the agent’s response later at `cli/src/commands/acp.rs:845-848`. Because the autoreply forwarder thread immediately writes to `stdout` (`cli/src/commands/acp.rs:350-378`), the IDE can observe the proxy-generated `session/prompt` before it ever receives the agent’s `stopReason` response that triggered it. That reorders the conversation transcript and violates the spec’d UX. Gate the autoreply send (or at least the “forward_to_ide” branch) until after the original response is flushed to the IDE.

- Tests: the only regressions covered in this module are helper primitives (`parse_agent_type`, `AutoreplyMessage` serialization, `JsonRpcId` normalization at the bottom of `cli/src/commands/acp.rs`). None of the Phase‑1 behaviors—prompt rewriting, metadata fallbacks, PostResponse dispatch, autoreply loop limits—have automated coverage even though the testing strategy in `ops/current/event-dispatch-gap-analysis.md:829-846` calls for it. Adding focused unit tests around `handle_session_prompt()`/`handle_post_response()` would prevent regressions in the multi-text handling and stopReason bookkeeping highlighted above.

Briefly: the proxy hits most of the checklist, but the multi-text rewrite gap, missing cleanup of tracked prompts, and the ordering race on autoreplies all contradict the implementation plan and will surface as confusing runtime behavior. Next steps would be to address those three code issues and add the missing unit/integration tests listed in the plan before declaring Phase 1 truly complete.
