Findings**
- `cli/src/handlers.rs:186-243` always returns `HookResponse::success()` even for `FlowResult::FailedBlock`, so `on_failure: block` in PrePrompt flows can never actually block a prompt. The ACP proxy still calls `response.is_blocking()` (cli/src/commands/acp.rs:1431-1435), but `exit_code` never becomes 2, so validation failures silently fall back to the original prompt. Please restore the previous behavior of emitting `HookResponse::blocking_failure(...)` on `FailedBlock`, otherwise flow authors have no way to enforce hard stops before the agent sees a bad prompt.

- PrePrompt context is dropped whenever flows also emit messages. `build_agent_context` only formats `response.messages` (cli/src/handlers.rs:126-142), and ACP builds the prompt as `agent_context + original_text`, only falling back to `response.context` when `agent_context` is empty (cli/src/commands/acp.rs:1438-1445). If a flow both logs a warning and adds context, the warning makes `agent_context` non-empty, so the prepended context is lost and the agent sees only the warning + original prompt. The plan explicitly called out combining messages and context; ACP needs to append `response.context` (or the structured prepend/append once that lands) instead of treating the two sources as mutually exclusive.

- PostResponse autoreplies suffer from the same problem. In `handle_post_response` the ACP proxy picks `build_agent_context` whenever it contains anything and ignores `response.context`, falling back to the latter only if no messages were emitted (cli/src/commands/acp.rs:1540-1547). This means any PostResponse flow that both raises a warning/error and sets an autoreply will end up sending the warning text as the autoreply, discarding the actual instructions meant for the agent. The proxy should merge messages and the autoreply context rather than picking one or the other.

- `ContextContent::Explicit` only accepts `Option<String>` for `prepend`/`append` (cli/src/flows/types.rs:205-221), so YAML snippets that use the documented array form (`prepend: ["line 1", "line 2"]`) or any representation that should deserialize into `TextLines::Multiple` can no longer be parsed—serde will error because a sequence cannot be converted to `String`. The new `ContextChunk`/`TextLines` machinery is therefore unreachable from flows, and only single-string prepends/appends work. `ContextContent` (and the corresponding `AutoreplyContent`) needs to use `Option<TextLines>` so both scalar and list forms remain valid, as shown in the tests under `cli/src/flows/context.rs:461-520`.
