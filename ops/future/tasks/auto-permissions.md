# Auto-Permissions for Aiki Commands

**Status**: Future Idea  
**Related**: Task System, Flows, aiki init

---

## Motivation

Currently, agents must request permission every time they run an `aiki` command (task add, task list, task run, etc.). This creates friction and interrupts the workflow, especially for the task system which is designed to be used frequently.

---

## Problem

When an agent tries to use the task system:

```bash
# Agent runs:
aiki task list

# Triggers shell.permission_asked event
# User must click "Allow" in the UI
# Then shell.completed fires with output
```

This makes the task system feel heavyweight and interrupts agent flow. For code reviews or automated task creation, this permission prompt happens repeatedly.

---

## Proposed Solution

**During `aiki init`, automatically configure permissions to allow `aiki` commands without prompting.**

Create `.aiki/flows/permissions.yml` with default allow rules:

```yaml
# .aiki/flows/permissions.yml
# Auto-generated by aiki init - customize as needed

shell.permission_asked:
  - name: "Allow aiki task commands"
    if: command starts_with "aiki task"
    then: allow
    
  - name: "Allow aiki review commands"
    if: command starts_with "aiki review"
    then: allow
    
  - name: "Allow read-only jj commands"
    if: |
      command starts_with "jj log" ||
      command starts_with "jj diff" ||
      command starts_with "jj show"
    then: allow
    
  - name: "Allow aiki doctor"
    if: command starts_with "aiki doctor"
    then: allow

# Everything else still requires permission (ask)
```

---

## Implementation

### 1. Add to `aiki init`

```rust
fn setup_default_permissions(repo_root: &Path, quiet: bool) -> Result<()> {
    let flows_dir = repo_root.join(".aiki/flows");
    fs::create_dir_all(&flows_dir)?;
    
    let permissions_file = flows_dir.join("permissions.yml");
    
    if !permissions_file.exists() {
        fs::write(&permissions_file, DEFAULT_PERMISSIONS_TEMPLATE)?;
        if !quiet {
            println!("✓ Configured default permissions for aiki commands");
            println!("  File: .aiki/flows/permissions.yml");
        }
    } else if !quiet {
        println!("✓ Permissions file already exists");
    }
    
    Ok(())
}

const DEFAULT_PERMISSIONS_TEMPLATE: &str = r#"# Aiki Auto-Permissions
# Generated by aiki init - customize as needed
#
# This allows agents to use aiki commands without permission prompts.
# Edit this file to customize which commands are auto-allowed.

shell.permission_asked:
  - name: "Allow aiki task commands"
    if: command starts_with "aiki task"
    then: allow
    
  - name: "Allow aiki review commands"
    if: command starts_with "aiki review"
    then: allow
    
  - name: "Allow read-only jj commands"
    if: |
      command starts_with "jj log" ||
      command starts_with "jj diff" ||
      command starts_with "jj show"
    then: allow

# Everything else requires permission (ask)
"#;
```

### 2. Call from init.rs

```rust
pub fn run(quiet: bool) -> Result<()> {
    // ... existing init steps ...
    
    // Setup default permissions
    if !quiet {
        println!("\nConfiguring default permissions...");
    }
    setup_default_permissions(&repo_root, quiet)?;
    
    // ... rest of init ...
}
```

## Benefits

1. **Frictionless task system** - Agents can use `aiki task` commands immediately
2. **Better UX** - No repeated permission prompts for safe commands
3. **Customizable** - Users can edit `.aiki/flows/permissions.yml` to adjust
4. **Safe defaults** - Only allows aiki commands and read-only jj commands
5. **Transparent** - File is visible and documented, not hidden magic

---

## Security Considerations

**Safe commands to auto-allow:**
- `aiki task *` - Task system is designed for agent use
- `aiki review *` - Code reviews are read-only analysis
- `jj log`, `jj diff`, `jj show` - Read-only inspection
- `aiki doctor` - Diagnostic tool

**Commands that should still require permission:**
- `git push`, `git commit` - External state changes
- `jj new`, `jj describe` - Working copy changes
- File system operations (rm, mv, etc.)
- Network operations (curl, wget)
- Package installs (npm install, etc.)

The default permissions should be conservative - only allow commands that are:
1. Read-only, OR
2. Explicitly designed for agent use (like the task system)

---

## Alternative Approaches Considered

### Option 1: Global auto-allow for all aiki commands
**Rejected**: Too broad, doesn't allow fine-grained control

### Option 2: Prompt during init
**Rejected**: Adds friction to init process, most users want task system to "just work"

### Option 3: Use MCP protocol permissions
**Rejected**: Aiki-specific, needs to work across all agents (Claude, Cursor, etc.)

### Option 4: No auto-permissions, require manual setup
**Rejected**: Makes task system feel heavyweight and enterprise-y, not AI-first

---

## User Customization

Users can customize by editing `.aiki/flows/permissions.yml`:

```yaml
# Stricter - only allow task list/show (read-only)
shell.permission_asked:
  - if: command == "aiki task list" || command == "aiki task show"
    then: allow

# More permissive - allow all jj commands
shell.permission_asked:
  - if: command starts_with "aiki" || command starts_with "jj"
    then: allow

# Hybrid - auto-allow during business hours only
shell.permission_asked:
  - if: command starts_with "aiki task" && time.hour >= 9 && time.hour < 17
    then: allow
```

## Implementation Phases

**Phase 1**: Basic implementation
- Create permissions.yml during init
- Support simple `starts_with` matching
- Default allow for `aiki task` and `aiki review`

**Phase 2**: Enhanced matching
- Support regex patterns
- Support boolean expressions (AND, OR, NOT)
- Support environment variables in conditions

**Phase 3**: Advanced features
- Time-based permissions
- User confirmation for certain patterns
- Audit log of auto-allowed commands
